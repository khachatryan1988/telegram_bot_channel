# Техническая документация MVP Telegram-бота для розыгрыша Domus

## 1. Назначение проекта
MVP Telegram-бот для розыгрыша в канале **@domus_stores**.

Цель:
- проверить подписку пользователя на канал,
- выдать персональную referral-ссылку,
- учитывать только валидные приглашения,
- активировать участие после 3 подтвержденных referrals,
- дать администратору возможность вручную выбрать победителя,
- поддержать повторный розыгрыш, если победитель не ответил в течение 48 часов.

---

## 2. Область MVP
В MVP входят:

- один Telegram-бот;
- один Telegram-канал: `@domus_stores`;
- один язык интерфейса: армянский;
- одна referral-механика;
- ручной запуск выбора победителя;
- локальная БД SQLite;
- один активный розыгрыш в рамках простой схемы данных.

В MVP **не входят**:
- веб-админка,
- многоязычность,
- сложный антифрод,
- интеграции с CRM,
- автозапуск по расписанию,
- несколько каналов,
- несколько параллельных конкурсов.

## 3. Пользовательский сценарий

### 3.1. Вход
Пользователь открывает бота и нажимает кнопку **«Մասնակցել»**.

### 3.2. Проверка участия
Бот показывает:
- кнопку перехода в канал,
- кнопку **«Ստուգել մասնակցությունս»**.

### 3.3. Успешная проверка
Если пользователь подписан на канал:
- бот подтверждает участие,
- формирует персональную referral-ссылку,
- показывает текущий прогресс.

### 3.4. Referral-механика
Каждый пользователь получает уникальную ссылку вида:

`https://t.me/<bot_username>?start=ref_<ref_code>`

Новый пользователь засчитывается как referral только если:
1. вошел по этой ссылке,
2. отправил `/start`,
3. успешно прошел проверку подписки.

### 3.5. Активация участия
После достижения порога:
- `3/3`

пользователь получает статус активного участника розыгрыша.

### 3.6. Выбор победителя
Администратор вручную запускает команду выбора победителя.

Победитель должен:
- оставаться подписчиком канала,
- ответить в течение 48 часов.

Если ответа нет:
- администратор запускает повторный выбор победителя.

## 4. Бизнес-правила

### 4.1. Проверка подписки
Пользователь считается верифицированным только если Telegram API возвращает статус участника канала.

Подходящие статусы:
- `member`
- `administrator`
- `creator`

Не подходят:
- `left`
- `kicked`

### 4.2. Учет referral
Referral засчитывается только один раз на одного приглашенного пользователя.

Ограничения:
- самореферал запрещен;
- один пользователь может быть привязан только к одному пригласившему;
- изменение referrer после первичной фиксации не допускается.

### 4.3. Eligibility
Пользователь становится участником розыгрыша, если:
- подписан на канал,
- прошел verify,
- имеет минимум 3 валидных verified referrals.

### 4.4. Победитель
Победитель выбирается случайно только среди пользователей со статусом `eligible`.

Перед финальным подтверждением победителя бот или администратор должны дополнительно проверить:
- что победитель все еще подписан на канал.

### 4.5. Таймаут победителя
После выбора победителя:
- фиксируется дедлайн 48 часов.

Если в течение 48 часов победитель не подтвердил себя ответом в бот:
- его статус меняется на `expired`,
- администратор может выполнить `re-draw`.

## 5. Роли

### 5.1. Пользователь
Может:
- войти в бота,
- пройти проверку подписки,
- получить referral-ссылку,
- видеть прогресс,
- подтвердить отклик в случае победы.

### 5.2. Администратор
Может:
- просматривать статистику,
- запускать выбор победителя,
- запускать повторный выбор,
- смотреть список участников.

## 10. Логика работы

### 10.1. `/start`
Сценарий:
1. Бот получает пользователя.
2. Если пользователь новый — создает запись.
3. Если в deep-link есть `ref_<code>`:
   - ищет referrer,
   - проверяет, что это не self-referral,
   - сохраняет связь, если она еще не была установлена.
4. Показывает стартовое сообщение и кнопку участия.

### 10.2. Verify
Сценарий:
1. Пользователь нажимает `Ստուգել մասնակցությունս`.
2. Бот вызывает проверку членства в `@domus_stores`.
3. Если подписка есть:
   - ставит `is_verified = true`,
   - при необходимости подтверждает referral у пригласившего,
   - пересчитывает прогресс.
4. Если прогресс >= 3:
   - ставит `is_eligible = true`.

### 10.3. Progress
Прогресс считается по числу уникальных приглашенных пользователей, которые:
- привязаны к данному referrer,
- прошли verify.

### 10.4. Draw
Сценарий:
1. Администратор запускает `/draw`.
2. Из БД берется список `eligible` пользователей.
3. Случайным образом выбирается один пользователь.
4. Перед окончательной фиксацией бот повторно проверяет подписку.
5. Если подписка подтверждена:
   - создается запись в `draws`,
   - победителю отправляется сообщение,
   - устанавливается дедлайн +48 часов.

### 10.5. Redraw
Сценарий:
1. Администратор запускает `/redraw`.
2. Предыдущая запись переводится в `expired`, если срок вышел.
3. Из выборки исключается истекший победитель текущего раунда.
4. Запускается новый случайный выбор.

## 11. Ограничения MVP
- нет автообнаружения отписки в реальном времени;
- нет полноценного антифрод-анализа;
- нет панели управления вне Telegram;
- нет cron-автоматизации;
- нет поддержки параллельных розыгрышей;
- нет экспорта в Excel в первой версии.

- `BOT_TOKEN`
- `CHANNEL_USERNAME=@domus_stores`
- `ADMIN_IDS=123456789,987654321`
- `REFERRAL_TARGET=3`
- `WINNER_RESPONSE_HOURS=48`
- `DB_PATH=bot.db`

---

## 13. Критерии приемки MVP
MVP считается рабочим, если:

1. новый пользователь может зайти в бота;
2. бот корректно проверяет подписку на `@domus_stores`;
3. бот выдает персональную referral-ссылку;
4. referral засчитывается только после verify приглашенного;
5. после 3 валидных referrals пользователь получает статус участника;
6. админ может вручную выбрать победителя;
7. победителю отправляется уведомление;
8. бот хранит дедлайн 48 часов;
9. возможен повторный выбор победителя через `/redraw`.
